UNIX was created to be a multiuser, multithreaded, secure operating system, and many UNIX-based OSs followed. Linux is available in many flavors — the Linux kernel is usually packaged with other software components to create a complete distribution.

Understanding Linux file structures is critical for forensic examiners because Linux systems are common on servers, embedded devices, and mobile platforms. Key Linux commands every examiner should know include:

Essential Linux Commands

uname –a displays system information. pwd shows the present working directory. ls –l lists files in long format showing permissions, owner, and timestamps. ls –ul filename displays the last access time for a file. ifconfig shows network interface configuration.

Core Top-Level Directories

A Linux system organizes files into a hierarchical directory tree starting from the root (/). Key directories include /bin (essential binaries), /etc (configuration files), /home (user directories), /var (variable data like logs), /dev (device files), and /tmp (temporary files). Understanding where specific types of data reside helps examiners locate evidence efficiently.

Forensic Tip: The /var/log directory contains system logs that are invaluable during investigations. User home directories (/home/username) often contain browser history, documents, and application data relevant to a case.

The Linux file system ext2fs and its journaling version ext3fs use inodes, metadata, and data as part of the file structure. Ext4 added support for partitions larger than 16 TB and is now considered the standard file system for most Linux distributions.

Four Components of the File System

Boot block — Contains the bootstrap code, the instructions for starting the system. Superblock — Specifies disk geometry, available space, and keeps track of all inodes. The superblock manages the file system and is considered part of the metadata. Inode blocks — Contain the first data after the superblock and are assigned to every file allocation unit. Data blocks — Where directories and files are actually stored on the disk drive, linked directly to inodes.

Evolution of Ext File Systems

Ext2 was the original Linux file system. Ext3 improved upon it by adding journaling as a built-in file recovery mechanism — the journal records changes before they are committed, enabling crash recovery. Ext4 further enhanced the system with support for larger partitions, improved management of large files, and a more flexible approach to adding file system features.

Key Concept: The superblock is the single most important metadata structure in a Linux file system. If the superblock is corrupted, the entire file system may become unreadable. Linux maintains backup copies of the superblock for recovery purposes.

Inodes (information nodes) are a key part of the Linux file system. They contain descriptive file or directory metadata and also link data stored in data blocks. The main purpose of inodes is to link data stored on data blocks.

Inode Contents

An assigned inode contains: the mode and type of the file or directory, the number of links, the UID and GID of the owner, the number of bytes, the file’s last access time and last modified time, the inode’s last status change time, block addresses for the file data, indirect, double-indirect, and triple-indirect block addresses, the current usage status, the number of actual blocks assigned, and the file generation number.

Indirect Pointers

UNIX/Linux file systems use inodes that implement direct, indirect, double-indirect, and triple-indirect pointers. Direct pointers reference data blocks directly. When files are too large for direct pointers, indirect pointers reference blocks that contain additional pointers to data blocks, creating a tree-like structure that can address very large files.

Bad Block Inode

The bad block inode keeps track of a disk’s bad sectors. Forensically, bad blocks can be used to hide information. The badblocks command (requires root) can find bad blocks. Other commands: mke2fs and e2fsck include safeguards that prevent overwriting important information.

Forensic Alert: Suspects may deliberately mark good sectors as bad blocks to hide data. Always examine the bad block inode during an investigation to check for hidden information.

A hard link is a pointer that allows accessing the same file by different filenames. All hard-linked filenames refer to the same inode and the same physical location on a drive. Use the ln command to create a hard link — all files pointing to the same inode must be on the same physical drive.

Link Count

Inside each inode is a field called link count that specifies the number of hard links. When the link count reaches zero, the file system can reclaim the data blocks.

Symbolic Links

Symbolic links (also called soft links or symlinks) are simply pointers to other files and are not included in the link count. Unlike hard links, symbolic links can point to items on other drives or other parts of the network. They have their own inode, separate from the target. Symbolic links depend on the existence of the destination they point to — if the target is deleted, the symbolic link becomes broken. Use ln –s to create symbolic links.

Forensic Significance: Understanding links is crucial for forensic analysis. Hard links mean that deleting one filename does not delete the data if other hard links exist. Symbolic links can reveal connections between files, directories, and even network locations used by a suspect.

macOS has evolved through several file systems. Before OS X, Apple used the Hierarchical File System (HFS), which stored files in nested directories. HFS+ (Extended Format File System) was introduced with Mac OS 8.1, supporting smaller file sizes on larger volumes for more efficient disk use. Apple File System (APFS), introduced with macOS High Sierra, writes metadata alongside data to help with crash protection.

Data Forks and Resource Forks

In older versions of macOS, a file consists of two parts: the data fork (containing the actual file data) and the resource fork (containing file metadata and application information such as menus, dialog boxes, icons, and executable code).

Volumes, Blocks, and Clumps

A volume is any storage medium used to store files. Logical blocks are collections of data that cannot exceed 512 bytes. Allocation blocks are sets of consecutive logical blocks assembled when a file is saved. Clumps are groups of contiguous allocation blocks used to reduce fragmentation.

Mac File System Components

Key components include: Boot blocks, Master Directory Block (MDB) or Volume Information Block (VIB) in HFS+, Volume Control Block, Extents overflow file (used when a file’s block list becomes too long for the catalog), Volume Bitmap, and the Catalog (maintaining relationships between all files and directories).

B*-Tree Structure

The B*-tree organizes the directory hierarchy and file block mapping. Actual file data is stored on leaf nodes. The B*-tree also uses header, index, and map nodes to maintain its structure.

Key Concept: APFS represents a significant modernization for Apple, with features like space sharing, cloning, snapshots, and native encryption — all of which have important forensic implications.

There are important differences between Linux and macOS file systems. Linux uses /home/username and /root directories; in macOS, the corresponding folders are /users/username and /private/var/root. The /home directory exists in macOS but is empty. macOS users have limited access to other accounts’ files, and the guest account is disabled by default.

Application Settings & Evidence

Application settings are stored in three formats: plaintext, plist files (preference files for installed applications), and SQLite databases. Plist files are particularly valuable for investigators as they record application preferences and usage data.

FileVault Encryption

FileVault, introduced with OS X 10.3, encrypts and decrypts a user’s /users directory. FileVault 2 was later introduced to address security vulnerabilities found in the original version, providing full-disk encryption.

Keychains & Unified Logging

Keychains manage passwords for applications, websites, and system files — the Keychain Access application can restore passwords. Unified logging is located in /var/db/diagnostics and /var/db/uuid.text, offering three utilities: log, log collect, and log show for forensic examination.

Acquiring a Mac Image

To examine a Mac, you need to make an image of the drive. Physical extraction can be challenging — removing drives from Mac Mini or MacBook Air cases requires special Apple screwdrivers. You need a macOS-compatible forensic boot CD/DVD to create the image. Turning off the mount function (Disk Arbitration) allows connecting a suspect drive without a write-blocking device.

Important: Deleted files on macOS go to the Trashes folder. However, if a file is deleted at the command line, it does NOT show up in the Trash — a critical detail for forensic examiners to understand.

Most commercial computer forensics tools can analyze Ext2, Ext3, Ext4, ReiserFS, and Reiser4 file systems. Freeware tools include Sleuth Kit and its web browser interface, Autopsy Forensic Browser.

Sleuth Kit & Autopsy

Sleuth Kit must be installed before Autopsy Browser, or Autopsy won’t be installed correctly. Together, they provide a powerful open-source forensic analysis platform capable of timeline generation, keyword searching, file recovery, and hash analysis on Linux file system images.

Foremost — File Carving Tool

Foremost is a freeware carving tool that can read many image file formats. It works by identifying file headers and footers to recover files from unallocated space. Its configuration file (foremost.conf) defines the file types and signatures to search for.

Tarballs

A tarball is a highly compressed data file containing one or more files or whole directories and their contents. Forensic examiners may encounter tarballs when analyzing Linux systems, as they are commonly used for backups and software distribution.

Kali Linux

Kali Linux includes several forensics tools in its arsenal. It is a purpose-built distribution for penetration testing and digital forensics, with tools for disk analysis, memory forensics, network analysis, and more.

Best Practice: When analyzing Linux file systems, use multiple tools and cross-reference your findings. Sleuth Kit provides command-line precision, Autopsy offers visual analysis, and Foremost excels at data carving from unallocated space.

While Linux and macOS share UNIX roots, their file systems and forensic considerations differ significantly:

File System Differences

Linux primarily uses Ext4 with inodes and superblocks. macOS uses APFS (or HFS+ on older systems) with a B*-tree catalog structure, data/resource forks, and allocation blocks. Both systems use hierarchical directory structures but organize user data differently.

Forensic Tool Availability

Linux forensics benefits from many open-source tools (Sleuth Kit, Foremost, Kali Linux toolset). macOS forensics requires specialized tools — some commercial tools like EnCase and X-Ways Forensics can analyze raw Mac images, while others are Mac-only solutions. Recon for macOS provides specialized Mac forensic capabilities.

Encryption Considerations

Linux systems may use LUKS (Linux Unified Key Setup) for disk encryption. macOS uses FileVault 2 for full-disk encryption. Both present challenges for forensic acquisition and require proper handling to obtain decrypted data when possible.

Summary: Whether examining a Linux server or a MacBook, understanding the specific file system structures, knowing where evidence is stored, and using the right tools for each platform is essential for a thorough forensic investigation.
