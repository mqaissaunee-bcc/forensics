Current Digital Forensics Tools

Evaluate, select, and validate the software and hardware tools used by digital forensics professionals to acquire, verify, extract, reconstruct, and report on digital evidence.

When selecting forensic tools, look for software that is versatile, flexible, and robust. Consider open-source tools alongside commercial options — the goal is to find the best value for as many features as possible.

Key Evaluation Questions

When evaluating forensic software, ask: On which OS does the tool run? Is the tool versatile enough for multiple tasks? Can it analyze more than one file system? Can a scripting language be used to automate repetitive functions? Does it have automated features? What is the vendor’s reputation for product support?

Two Major Categories

Digital forensics tools fall into two broad categories: Hardware (ranging from simple single-purpose components to complete computer systems and servers) and Software (either command-line or GUI-based).

Tip: Read as much information as you can about a product and compare it with similar tools before making a purchase decision. Look for independent reviews and community feedback.

NIST’s Computer Forensics Tool Testing (CFTT) program and ISO standard 27037 state that Digital Evidence First Responders (DEFRs) should use validated tools. Digital forensics tool functions are divided into five major categories:

1. Acquisition

Acquisition is the task of making a copy of the original drive. Subfunctions include physical data copy (entire drive), logical data copy (partition only), data acquisition format (raw vs. proprietary), command-line acquisition, GUI acquisition, and remote, live, and memory acquisitions.

2. Validation & Verification

Validation confirms a tool is functioning as intended. Verification proves that two sets of data are identical by calculating hash values. Subfunctions include hashing (CRC-32, MD5, SHA-1), filtering (based on hash value sets), and analyzing file headers to discriminate files by type.

The National Software Reference Library (NSRL) has compiled a list of known file hashes for a variety of OSs, applications, and images.

3. Extraction

Extraction is the recovery task and the most challenging to master. Subfunctions include data viewing, keyword searching, decompressing, carving, decrypting, and bookmarking/tagging.

4. Reconstruction

Reconstruction re-creates a suspect drive to show what happened during a crime. Techniques include disk-to-disk copy, partition-to-partition copy, image-to-disk copy, image-to-partition copy, disk-to-image copy, and rebuilding files from data runs and carving.

5. Reporting

Forensic tools produce reports with subfunctions including bookmarking/tagging, log reports, timelines, and report generators that output in a variety of electronic formats.

Important: Encrypted files and systems are a major challenge. Password recovery tools can generate potential password lists for a password dictionary attack. If that fails, a brute-force attack tries every combination of characters.

The first tools that analyzed and extracted data from floppy disks and hard disks were MS-DOS tools for IBM PC file systems. Norton DiskEdit was one of the earliest MS-DOS tools used for computer investigations.

One advantage of command-line tools is that they require few system resources because they are designed to run in minimal configurations. Some command-line forensics tools are created specifically for Windows CLI platforms, while others exist for macOS and Linux.

Key Advantage: Command-line tools are lightweight, fast, and can be scripted for repeatable automated tasks — ideal for large-scale investigations or environments with limited computing resources.

Linux is gaining popularity in forensics due to cost advantages and improving GUI features. Key Linux forensic tools include:

SMART

SMART was designed to be installed on numerous Linux versions. It can analyze a variety of file systems and includes several plug-in utilities and a hex viewer.

Helix

Helix is one of the easiest suites to begin with. You can load it on a live Windows system, and it loads as a bootable Linux OS from a cold boot.

Kali Linux

Kali Linux (formerly BackTrack) includes a variety of forensics tools and has an easy-to-use KDE interface.

Autopsy & Sleuth Kit

Sleuth Kit is a Linux forensics tool, and Autopsy is the browser interface used to access Sleuth Kit’s tools. These are powerful open-source options.

Forcepoint Threat Protection

Formerly known as Second Look, this is a Linux memory analysis tool designed for both onsite and remote (enterprise) memory acquisition to determine whether malware is present.

Note: A live acquisition has not been accepted as a valid forensics practice by some international courts. Always verify what is admissible in your jurisdiction.

GUI forensics tools simplify investigations and help train beginning investigators. Most come in suites of tools that bundle multiple functions together.

Advantages of GUI Tools

Ease of use — intuitive interfaces reduce the learning curve. Multitasking — run multiple analyses simultaneously. No need to learn older OSs — modern interfaces abstract away complexity.

Disadvantages of GUI Tools

Excessive resource requirements — GUI tools demand more memory and processing power. Inconsistent results — some tools may produce varying output. Tool dependencies — relying on a single tool suite can create blind spots.

Other Considerations

When planning your lab, consider flexibility, reliability, and expandability. Keep a library with older versions of your tools as a backward compatibility measure.

Tip: GUI forensic tools are no different from other GUI tools when compared with their equivalent command-line utilities. Always verify GUI results with an independent tool.

Hardware will eventually fail, so you need to carefully plan for equipment replacement. Budget for failures, maintenance fees, and anticipated replacements.

Forensic Workstations

Three types: Stationary (full lab setups), Portable (field work), and Lightweight (minimal configurations). You need to balance what you need and what your system can handle.

Build vs. Buy

Building your own: Customized to your needs and can save money, but support is harder to find and costs can spiral if not careful. Buying from a vendor (e.g., F.R.E.D. units): Vendor support saves time and frustration when problems arise.

Workstation Recommendations

Key recommendations include: USB 3.0 and FireWire data acquisition capabilities, expansion devices, as much memory and processing power as budget allows, a power supply with battery backup, extra cables, a SCSI controller card, external FireWire and USB ports, drive adapter bridges, ergonomic peripherals, and a high-end video card with at least a 17-inch monitor.

Budget Tip: If you have a limited budget, high-end gaming PCs offer excellent processing power and graphics capabilities that translate well to forensic workstation needs.

A write-blocker prevents data writes to a hard disk — essential for preserving evidence integrity.

Software vs. Hardware Write-Blockers

Software-enabled blockers typically run in a shell mode (Windows CLI). Hardware options are ideal for GUI forensic tools and act as a bridge between the suspect drive and the forensic workstation.

How They Work

When you use a write-blocker, you can navigate to the blocked drive with any application, but you cannot write to the drive. The write-blocker discards the written data. However, for the OS, the data copy appears successful.

Connecting Technologies

Write-blockers support various connections: FireWire, USB 2.0 and 3.0, and SATA, PATA, and SCSI controllers.

Critical Rule: Write-blocking devices or utilities must be used with GUI acquisition tools. Never connect a suspect drive to your forensic workstation without a write-blocker in place.

Preserving the integrity of your evidence is critical, especially if it is admitted to court. You must be able to demonstrate that your tools are reliable and produce accurate results.

NIST’s CFTT Program

NIST’s Computer Forensics Tool Testing (CFTT) project manages research on digital forensics tools. NIST has created criteria based on standard testing methods and ISO 17025 criteria for testing items with no current standards.

Lab Testing Criteria

Your lab must: Establish categories for digital forensics tools, identify category requirements, develop test assertions, identify test cases, establish a test method, and report test results. ISO 5725 specifies results must be repeatable and reproducible.

NSRL & Reference Data Set

The National Software Reference Library (NSRL) collects all known hash values for commercial software applications and OS files. It uses SHA-1 to generate a known set of digital signatures called the Reference Data Set (RDS). RDS can be used to filter known information and identify known bad files.

Validation Protocols

Always verify results using at least two different tools. Use a disk editor (such as Hex Workshop or WinHex) to compare results. The examination protocol: conduct investigation with a GUI tool, verify with a disk editor, compare hash values from both tools.

Upgrade Protocol: Test new releases, OS patches, and upgrades. If a problem is found, do not use the tool until the vendor fixes it. Use a test hard disk for validation and check the web for updates and validation tests.
